# SupportPilot AI UI 需求文档

## 1. 文档目标

本文档用于指导 SupportPilot AI 下一阶段前端 UI 调整，目标是把当前偏“电商售后演示后台”的界面，调整为更通用的企业客服运营平台。

本次 UI 需求重点不是重写业务链路，而是明确：

- 产品定位如何表达
- 官网首页应该有哪些区块
- 工作台页面如何命名和组织
- 视觉风格应该如何适配 B 端 SaaS
- 当前接口和数据如何继续使用
- 哪些能力先做 UI 占位，哪些需要后续后端补接口

## 2. 当前项目情况

### 2.1 已有能力

当前项目已经具备：

- FastAPI 后端
- Streamlit 前端
- RAG 知识库检索
- AI 工作流编排
- trace 日志记录
- 基础运营指标接口
- 电商售后知识库示例数据
- 工作台页面：概览、队列、AI 记录、工单、知识库状态、开发调试

当前可用接口：

- `POST /api/chat`
- `POST /api/chat/stream`
- `GET /api/metrics`
- `GET /api/traces`

### 2.2 当前问题

- UI 仍然强绑定“电商售后”定位。
- 缺少对外展示官网首页。
- 工作台导航命名不够统一。
- `开发调试` 目前作为一级页面展示，不适合正式产品结构。
- `知识库状态` 目前只是只读状态页，后续需要升级为轻量知识库管理页。

## 3. 产品定位

产品名称：

- `SupportPilot AI`

产品定位：

- 企业客服运营平台

当前版本表达：

- 前台官网表达为通用客服平台。
- 工作台表达为企业客服运营后台。
- 当前已有电商售后数据继续作为示例数据存在。
- 不在产品界面上强调“内置电商售后示例知识包”，这部分用于项目讲解和文档说明。

后续演进方向：

- 当前：单平台 MVP
- 下一阶段：支持知识库管理、更多业务场景配置
- 后续：多企业、多行业、多知识库、多工作流配置

## 4. 信息架构

### 4.1 官网首页

- 对外展示官网首页
- 不是企业内部系统入口页
- 不是纯登录页

- 说明产品是什么

- 展示核心功能模块
- 展示适用场景
- 说明基本处理流程
- 提供进入工作台的入口
- 提供 AI 产品问答窗口

首页顶部导航：

- `AI Agent`
- `产品`
- `行业`
- `帮助`
- 右侧：`免费试用`、`登录`

顶部导航二级标签：

`AI Agent`

- `概览`
- `文本AI`
- `语音AI`

`产品`

- `智能分流`
- `知识库`
- `人工协同`
- `服务工单`
- `数据分析`

`行业`

- `电商零售`
- `教育咨询`
- `本地服务`
- `企业服务`
- `生活消费`

`帮助`

- `帮助文档`
- `接入指南`
- `常见问题`

首页区块顺序：

1. 顶部导航区

3. 产品亮点区
4. 知识库介绍区

9. 底部区

### 4.2 工作台

工作台定位：

- B 端客服运营后台
- 承接真实业务操作和数据查看
- 不承担官网宣传功能

一级导航要求：

- 尽量控制在四个字以内
- 命名清晰、克制、偏真实后台

最终一级导航：

- `数据概览`
- `人工队列`
- `处理记录`
- `服务工单`
- `知识库`

调试能力：

- `开发调试` 不再作为一级导航。
- 调试内容下沉到底部折叠区。
- 当前阶段可以保留，后续正式化时可以删除或隐藏。

## 5. 官网首页 UI 需求

### 5.0 参考页面适配原则

参考页面：

- 美洽 AI Agent 页面：`https://www.meiqia.com/ai-agent#text-ai`

可借鉴：

- 首页标题层级清晰。
- 先讲 AI Agent 核心能力，再讲知识库、场景接入和具体客服能力。
- 区块标题短、直接，适合 SaaS 产品官网。
- 页面从产品价值逐步过渡到能力模块。

不照搬：

- 不照搬文案。
- 不照搬品牌视觉。
- 可以保留 `免费试用` 和 `登录` 作为官网入口，但当前版本不实现真实注册和权限系统。
- 不加入客户案例。
- 语音AI只能作为产品展示或模拟流程，不作为当前真实语音机器人能力。

本项目适配方式：

- 保留 `产品亮点`、`知识库介绍`、`全渠道接入`、`语音AI介绍`、`服务闭环` 五个核心展示区。
- 把参考页偏商业转化的内容，改为适合 MVP 的产品能力说明。
- 首屏按当前方案保留主文案、小字说明、主视觉，以及 `在线咨询` 和 `免费试用` 两个按钮。

### 5.1 顶部导航区

放置元素：

- 产品 Logo 或文字标识
- 产品名：`SupportPilot AI`
- 导航标签：`AI Agent`、`产品`、`行业`、`帮助`
- 右侧按钮：`免费试用`、`登录`

交互要求：

- `AI Agent`：展开二级菜单，包含 `概览`、`文本AI`、`语音AI`
- `产品`：展开二级菜单，包含 `智能分流`、`知识库`、`人工协同`、`服务工单`、`数据分析`
- `行业`：展开二级菜单，包含电商、教育、本地服务、企业服务、生活消费等场景
- `帮助`：展开二级菜单，包含 `帮助文档`、`接入指南`、`常见问题`
- `免费试用`：进入工作台入口页
- `登录`：进入工作台入口页

产品二级标签交互边界：

- `智能分流`、`知识库`、`人工协同`、`服务工单`、`数据分析` 只跳转到官网内对应功能介绍区或弹出功能说明。
- 不进入工作台业务详情页。
- 不打开真实业务数据页面。
- 真正进入业务页面只能通过 `免费试用` / `登录` 到工作台入口页，再点击工作台五个业务入口。

不需要：

- `联系管理员`
- `客户案例`

### 5.2 首屏主视觉区

首屏核心元素：

- 主标题
- 主标题下方一行小字说明
- 主视觉
- `在线咨询` 按钮
- `免费试用` 按钮

主标题文案方向：

- `面向企业服务场景的智能客服运营平台`

说明文案方向：

- 用一行小字说明平台统一承接客户咨询、AI 处理标准问题、人工接管复杂问题，并通过工单、知识库和数据完成服务闭环。

主视觉要求：

- 用于展示 AI 客服、人工协同、知识库和数据后台之间的关系。
- 可以是产品界面组合图、抽象流程图或产品场景图。
- 不使用纯装饰渐变图。

在线咨询悬浮窗：

- 官网首页的产品问答助手
- 用于回答产品定位、核心价值、适用行业、功能模块等问题
- 不是客服工作台里的客户会话窗口
- 在长页面右下角悬浮存在
- 点击 `在线咨询` 按钮时打开该悬浮窗

悬浮窗元素：

- 窗口标题
- 简短提示语
- 输入框
- `发送` 按钮

提示语方向：

- `可询问产品定位、核心能力与适用场景`

不需要：

- 状态标签
- 预设问题按钮
- 登录卡片

### 5.3 产品亮点区

该区块用于总结产品核心亮点，不做复杂功能堆叠。

区块标题建议：

- `产品亮点`

副标题文案方向：

- 用一段文案总结本产品的核心价值：AI 自动承接标准问题，复杂问题进入人工或工单，服务过程沉淀为可追踪的数据。

展示重点：

- `意图识别`
- `智能分流`
- `自动回复`
- `人工协同`
- `服务工单`
- `数据沉淀`

交互要求：

- 当前版本可以是文案 + 少量能力点展示。
- 不做工作台快捷入口。
- 不展示技术名词，如 LangGraph、trace、raw JSON。

### 5.4 知识库介绍区

该区块用于承接“AI 能力需要知识驱动”的产品逻辑，参考美洽页面中对知识库和知识训练能力的展示方式，但不要照搬其文案。

区块标题建议：

- `知识库`

副标题文案方向：

- 说明企业可以把服务政策、业务规则、常见问题沉淀为知识内容，供 AI 检索和客服协作使用。

展示内容建议：

- `知识沉淀`
- `内容检索`
- `引用依据`
- `持续维护`

交互要求：

- 当前版本可展示为静态能力说明。
- 后续与工作台 `知识库` 页面联动。
- 不在官网直接暴露“电商售后示例包”。

### 5.5 全渠道接入区

该区块用于展示未来可扩展的渠道接入能力。当前版本做展示和 mock 状态，不做真实第三方平台接入。

区块标题建议：

- `全渠道接入`

副标题文案方向：

- 说明平台可统一承接来自网页、小程序、App、企业微信、邮件等渠道的客户咨询。

渠道卡片建议：

- `网页客服`
- `小程序`
- `企业微信`
- `App`
- `邮件`

每张卡片包含：

- 渠道名称
- 接入状态：`已接入`、`配置中`、`未启用`
- 一句说明

要求：

- 不接真实第三方 API。
- 不展示真实渠道后台配置。
- 明确作为产品展示和后续扩展方向。

### 5.6 语音AI介绍区

该区块用于展示语音 AI 的产品规划能力。当前版本只做介绍或模拟流程，不做真实录音、ASR、TTS、电话接入。

区块标题建议：

- `语音AI`

副标题文案方向：

- 说明语音咨询未来可通过 ASR 转写、AI 理解、知识检索、TTS 回复形成语音客服流程。

展示内容建议：

- `语音输入`
- `ASR转写`
- `AI处理`
- `语音回复`

要求：

- 当前只做 mock 或静态展示。
- 不宣称已经完成真实语音机器人。
- 不展示呼叫中心、外呼系统等当前未实现能力。

### 5.7 服务闭环区

该区块替代原来的“处理流程区”，标题更偏产品价值，表达从客户咨询到数据沉淀的完整闭环。

区块标题建议：

- `服务闭环`

推荐流程：

1. 客户发起咨询
2. AI 识别问题并检索知识
3. 标准问题自动处理
4. 复杂问题进入人工或工单
5. 处理结果沉淀到数据后台

要求：

- 每步一句短文案。
- 讲业务流程，不讲技术实现。
- 不展示 LangGraph、RAG、trace、raw JSON 等技术细节。

### 5.8 免费试用转化区

该区块作为长页面结尾 CTA。

放置元素：

- 一段产品价值总结
- `免费试用` 按钮
- `在线咨询` 按钮

交互要求：

- 点击 `免费试用` 进入工作台入口页。
- 点击 `在线咨询` 打开首页右下角悬浮咨询窗。

### 5.9 帮助文档入口区

该区块只保留一个入口，不展开文档列表。

放置元素：

- 一句简短说明
- 按钮：`帮助文档`

不需要：

- 多个帮助卡片
- 常见问题列表
- `联系管理员`

### 5.10 底部区

放置元素：

- `SupportPilot AI`
- `工作台`
- `帮助文档`

要求：

- 保持简洁。
- 不放虚假客户案例、价格、法务页面或复杂链接。

## 6. 工作台 UI 需求

### 6.1 工作台入口页

进入来源：

- 官网顶部 `免费试用`
- 官网顶部 `登录`
- 官网首屏 `免费试用`
- 结尾 CTA `免费试用`

顶部导航栏：

- Logo
- `服务中心`
- `员工中心`
- `客服管理`
- `系统管理`

页面结构：

- 上半部分：工作台视觉区
- 下半部分：五个业务入口按钮

上半部分要求：

- 展示产品欢迎信息或工作台概览视觉。
- 不做官网营销文案。
- 可以展示平台名、当前企业、基础状态或简短说明。

下半部分五个入口：

- `数据概览`
- `人工队列`
- `处理记录`
- `服务工单`
- `知识库`

交互要求：

- 点击任一入口进入对应业务详情页。
- 当前版本可以不实现真实登录态。

### 6.2 业务详情页通用布局

业务详情页是工作台的第三层页面。

通用布局：

- 左侧栏顶部：当前大标签
- 左侧栏下方：二级标签
- 右侧主区域：当前模块内容

五个业务详情页都采用同一布局规则：

- `数据概览`
- `人工队列`
- `处理记录`
- `服务工单`
- `知识库`

### 6.3 全局布局

工作台应包含：

- 一级导航
- 主内容区
- 筛选或操作区
- 数据表格或详情区
- 底部折叠调试区

视觉要求：

- 像真实 B 端运营后台。
- 信息密度可以高，但不能拥挤。
- 优先支持扫描、筛选、查看详情、处理队列。
- 不做官网式大标题和营销式展示。

### 6.4 数据概览

页面名：

- `数据概览`

页面目的：

- 展示客服处理效率、AI 自动处理量、人工积压、工单量和风险问题。

关键数据：

- 咨询总量
- AI 自动处理量
- AI 处理率
- 转人工数量
- 工单数量
- 平均响应时间
- 问题分类分布
- 待处理或高风险问题

交互按钮：

- `今日`
- `近7天`
- `近30天`
- `刷新`
- `异常问题`
- `人工积压`

左侧二级标签：

- `今日数据`
- `趋势分析`
- `问题分布`
- `人工积压`
- `风险问题`

数据来源：

- `GET /api/metrics`
- `GET /api/traces`

### 6.5 人工队列

页面名：

- `人工队列`

页面目的：

- 展示 AI 无法完全处理、需要人工接管的问题。

列表字段：

- Trace ID
- 问题类型
- 风险等级
- 用户问题摘要
- 响应耗时或等待时长
- 建议动作
- 工单状态

页面筛选：

- `全部`
- `待接管`
- `高风险`
- `退款类`
- `投诉类`
- `刷新`

单条操作：

- `详情`
- `接管`
- `建工单`
- `已处理`
- `复制回复`

左侧二级标签：

- `待接管`
- `高风险`
- `退款类`
- `投诉类`
- `已处理`

数据来源：

- `GET /api/traces`

筛选规则：

- 展示 `action == "handoff"` 的记录。

### 6.6 处理记录

页面名：

- `处理记录`

页面目的：

- 展示 AI 已处理过的问题、结果和流转去向。

列表字段：

- Trace ID
- 问题类型
- 原始问题摘要
- 处理结果
- 动作类型
- 响应耗时
- 命中知识来源或引用数量

页面筛选：

- `全部`
- `自动解决`
- `转人工`
- `建工单`
- `类型筛选`
- `刷新`

单条操作：

- `摘要`
- `依据`
- `流转`

左侧二级标签：

- `全部记录`
- `自动解决`
- `转人工`
- `建工单`
- `异常记录`

数据来源：

- `GET /api/traces`

### 6.7 服务工单

页面名：

- `服务工单`

页面目的：

- 展示需要结构化跟进的问题。

列表字段：

- 工单 ID
- 问题类型
- 优先级
- 状态
- 负责人
- 来源问题
- 创建时间或处理耗时

页面筛选：

- `全部`
- `待处理`
- `处理中`
- `已完成`
- `高优先`
- `刷新`

单条操作：

- `详情`
- `改状态`
- `指派`
- `备注`
- `完成`

左侧二级标签：

- `全部工单`
- `待处理`
- `处理中`
- `已完成`
- `高优先`

数据来源：

- `GET /api/traces`

筛选规则：

### 6.8 知识库

页面名：

- `知识库`

页面目的：

- 从只读状态页升级为轻量知识库管理页。

当前版本要求：

- 继续展示已有知识文档。
- 不在 UI 上强调“电商售后示例包”。
- 以通用知识库管理页的形式组织界面。

列表字段：

- 文档名称
- 分类
- 状态
- 来源类型
- 最近更新时间
- 引用次数或使用情况

交互按钮：

- `新增`
- `查看`
- `编辑`
- `启用`
- `停用`
- `删除`

左侧二级标签：

- `全部文档`
- `已启用`
- `未启用`
- `引用记录`
- `知识配置`

实现边界：

- 用户界面不要写“demo 数据”。
- 开发说明或项目讲解中可以说明当前知识库管理能力仍处于 MVP 阶段。

后续接口建议：

- `GET /api/knowledge/documents/{document_id}`

## 7. 视觉风格要求

### 7.1 整体风格

产品应呈现真实企业 SaaS 工具气质。

关键词：

- 专业
- 克制
- 清晰
- 可信
- 高效
- 运营导向

避免：

- 过度营销化
- 大面积渐变
- 虚假客户案例
- 过多装饰图形
- 单一色系铺满页面
- 复杂 BI 大屏感
- 面试作品包装感

### 7.2 官网首页视觉

要求：

- 可以比工作台更有展示感，但仍然保持克制。
- 首屏重点是产品标题和 AI 产品问答窗口。
- 页面层级可参考成熟客服 SaaS 官网的组织方式：首屏定位、产品亮点、知识库、全渠道接入、语音AI介绍、服务闭环、免费试用转化。
- 首页标题应体现“产品能力递进”，不要只平铺功能列表。
- 区块之间要有明确视觉节奏，但不做夸张营销页。
- 不做登录卡片。
- 不做客户案例模块。
- 不做夸张营销文案。

参考页面吸收点：

- 可借鉴美洽 AI Agent 页面清晰的能力分层和大标题节奏。
- 不照搬其文案、品牌表达、客户案例、预约演示、登录注册等商业转化模块。
- 当前项目真实核心能力仍然是 AI 文本客服、知识库、人工协同、工单流转和数据沉淀。
- 全渠道接入和语音AI当前只做展示或 mock，不作为真实生产能力。

### 7.3 工作台视觉

要求：

- 偏后台工具，不偏宣传页。
- 表格、筛选、状态标签、详情区要清晰。
- 卡片圆角不超过 8px。
- 不要卡片套卡片。
- 不要大面积 hero 区块。
- 调试信息视觉上要弱化。

### 7.4 颜色与组件

推荐方向：

- 中性背景
- 白色或近白内容区
- 深色文字层级
- 蓝色或青绿色作为主操作色
- 红色用于高风险
- 黄色或橙色用于警告
- 绿色用于成功或已解决

组件要求：

- 状态用 badge 表示。
- 筛选用 tabs、segmented control 或短按钮组。
- 表格字段要可读。
- 空状态要说明当前无数据，不要显得系统错误。
- 按钮文案短而明确。

### 8.3 首页 AI 问答接口要求

首页 AI 对话窗口不应默认调用 `/api/chat`。

原因：

- `/api/chat` 是客户服务问题处理接口。
- 官网首页 AI 对话是产品问答助手。
- 两者数据不应该混入同一套 trace 和运营指标。

建议新增接口：

- `POST /api/product-chat`

请求示例：

```json
{
  "message": "这个产品适合哪些行业？"
}
```

响应示例：

```json
{
  "answer": "SupportPilot AI 适用于需要统一承接客户咨询、AI分流、人工协同和知识库管理的企业服务场景。"
}
```

MVP 处理方式：

- 第一版可先做前端 mock 回答。
- 不把官网问答写入 `traces.jsonl`。
- 不计入 `/api/metrics`。

## 10. 验收标准

### 10.1 官网首页验收

- 存在官网首页。
- 顶部导航包含 `AI Agent`、`产品`、`行业`、`帮助`，右侧包含 `免费试用`、`登录`。
- `AI Agent` 二级标签只包含 `概览`、`文本AI`、`语音AI`。
- `产品` 二级标签按产品功能组织，包含 `智能分流`、`知识库`、`人工协同`、`服务工单`、`数据分析`。
- `产品` 二级标签只做官网功能介绍，不跳转到工作台业务详情页。
- 首屏包含主标题、小字说明、主视觉、`在线咨询` 和 `免费试用`。
- 首屏没有状态标签。
- 首屏没有预设问题按钮。
- 首页包含产品亮点、知识库介绍、全渠道接入、语音AI介绍、服务闭环、免费试用转化和底部区。
- 首页不包含客户案例。
- 首页不包含注册、联系管理员、真实客户案例。

### 10.2 工作台验收

- 一级导航为 `数据概览`、`人工队列`、`处理记录`、`服务工单`、`知识库`。
- `开发调试` 不再作为一级导航。
- 产品级标题不再强绑定电商售后。
- 电商内容可以作为表格中的示例数据继续出现。
- `/api/metrics` 和 `/api/traces` 继续可用。

### 10.3 知识库验收

- 页面名称为 `知识库`。
- 页面不再只是只读状态展示。
- 页面结构预留 `新增`、`查看`、`编辑`、`启用`、`停用`、`删除`。
- 如果后端未实现，交互可以先 mock。

### 10.4 技术验收

- FastAPI 可启动。
- Streamlit 可启动。
- 现有测试继续通过。
- 现有 API 向后兼容。
- 首页 AI 产品问答不写入客服处理 trace。
- 首页 AI 产品问答不影响运营指标。
